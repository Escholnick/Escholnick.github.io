Week 2: Computer Vision Maze Navigator

Design Challenge

This week's design challenge was primarily a software one. The goal was to use the car chassis from the previous week, along with a camera, to let the robot navigate a maze autonomously, with our only input being neon page-markers placed in the maze for the camera to see.

General Approach

My group's general approach for this project was slightly complicated, but was designed to allow for maximum adaptability once the maze was revealed. The basic idea was to use a combination of two different-colored markers to create an arrow, then program the robot to turn to be in alignment with that arrow. Combined with nudge functions and a little bit of clever controller programming, hopefully the robot would be able to navigate the maze.

The idea of the arrow was to create a two-note T-shaped structure, with one note of an arbitrary color (let's say orange) as the body of the arrow, and a second, different colored note (let's say purple) placed perpedicularly to mark the head of the arrow. The code would detect the centers of the two notes, and from there draw a line from orange to purple. It would then turn so that the slope of the line was as steeply positive as possible ( and therefore the arrow was facing straight forward). It would then use nudge functions to center the arrow relative to itself, preform one last turn check to be safe, and return to moving forward.

There was also a secondary backup plan in place in case the more complicated idea did not work out. It was to do the basic thing and use different colors as different turn flags, such that blue meant "left 90", yellow meant "right 90" and so on. The idea was much simpler and easier to implement, but had less finesse and controll to it. As it would turn out, this would be the approach we fell back to on Wednesday when things started to go wrong.

Daily Work

Day 1

The majority of day 1 was spent on writing a basic controller class, which would set up the motors and could be used to drive and turn. The idea was to make things as easy to write as possible going forward. Most of the rest of the day was spent callibrating the controller so that the motors were relatively equal in relation to each other. This was to help prevent the possibility of the robot trying to drive straight but actually turning to one side as it went. The last half-hour or so was spent working on the basics of the computer vision element, setting up camera windows and such.

Day 2

Day 2 was spent on implementing the majority of the computer vision code, settting up color isolation filters and finding centroids. There was also a large amount of time dedicated to refactoring the code to be class based and multi-file rather than a mess of spagheti. Lots of work was completed, but there wasn't much variety to it.
There was also work on getting the computer vision data out and using it to turn, though that did not progress particularly well.

Day 3

Day 3 was largely spent bug-fixing and redoing the half-finished turning code so that there wasn't huge slow waits every time the robot tried to turn. We were able to fix that bug, but due to a prevlence of other bugs we switched to a backup script that turned at a variety of angles based on which colors it saw. The remainder of the day was spent testing and improving this backup script.
